## 1. 模块定位

应用模块是客户端使用计算能力的主要起点，承载“找应用、添加应用、选择提交方式、配置作业参数、发起作业或图形会话、进入命令行、处理软件购买/开通/到期”的完整链路。它连接应用市场、我的应用、标准作业提交、图形提交、命令行、文件选择、作业结果、软件服务和 SClaw 日志分析。

该模块主要用户包括普通计算用户、团队成员、管理员、购买过软件或服务的用户。核心价值是把复杂的软件、区域、资源、版本、提交方式和文件输入封装为可操作入口，让用户从应用卡片进入适合自己的计算场景。

## 9. 模块联动关系

- 应用模块是作业模块的入口：模板提交成功后进入作业结果，用户可跳转作业管理和作业详情。
- 应用模块与文件模块联动：提交表单中的输入文件、目录、上传、新建目录依赖文件选择器。
- 应用模块与 Shell 模块联动：命令行提交打开 Shell，并携带应用、规格、区域、模板等上下文。
- 应用模块与图形模块联动：图形提交和 CreateGraph 创建 VNC/DCV/RDP/VM/Frame 会话，后续进入图形会话列表或独立窗口。
- 应用模块与费用/商城联动：应用到期、软件未购买、商品续购和已购商品列表均跳转商城或商品详情。
- 应用模块与资源模块联动：无资源状态引导购买或刷新，提交按钮受资源、区域、节点、账号状态影响。
- 应用模块与帮助/客服联动：提交失败、版本损坏、RDP 未配置、服务群提示等场景提供客服入口。
- 应用模块与 SClaw 联动：作业日志失败分析可进入智能分析；应用卡片和提交表单自身未发现直接调用 SClaw 的入口。

## 12. 面向重构的产品建议

| 建议 | 说明 |
|---|---|
| 保持“应用 -> 提交方式 -> 作业/图形/Shell”的主链路清晰 | 应用是客户端发起计算任务的核心入口，重构时应优先保障各提交方式路径完整 |
| 将应用状态和商品状态统一建模 | 可更新、开通中、已到期、未开通、版本损坏等状态分散在卡片、提交页和商品页，建议统一状态字典 |
| 将提交前置条件集中展示 | 资源、区域、账号、软件、版本、参数、节点等条件共同影响提交，建议在提交页形成统一校验区 |
| 保留文件选择器与提交表单的低摩擦联动 | 文件输入是作业提交关键能力，需要继续支持上传、新建目录、路径回填 |
| 明确 Docker 与商品页面入口策略 | Docker 和我的商品功能代码完整度较高，需要产品确认公开入口和主导航关系 |
| 将图形创建入口与图形模块统一规则 | 应用侧创建图形和图形列表会话管理需共享状态、异常和命名 |
| 将申请开通、购买、续购归入统一“获取权限”路径 | 用户遇到无软件、已到期、未开通时需要一致的引导体验 |
| 将 SClaw 能力引入提交异常和日志分析 | 提交失败、软件版本异常、资源不足等场景适合由 AI 辅助解释和给出下一步 |

- 已确认功能：我的应用、应用中心、应用搜索/筛选/排序、添加/移除应用、提交方式选择、模板提交、命令行提交、图形创建、URL 打开、Docker 创建、文件选择、软件版本选择、提交成功/失败、无资源/无软件引导、软件更新、申请开通、已购商品、续购/购买、技术服务确认完成。
- 新增发现功能：服务群/客服提示、技术服务确认完成、文件/目录选择中的上传与新建目录、软件版本损坏/区域新版本提示、作业日志智能分析联动。
- 待确认功能：商品页面主入口、Docker 应用公开范围、应用卡片/提交表单与 SClaw 的直接联动边界。
